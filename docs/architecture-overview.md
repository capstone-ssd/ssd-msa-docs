# SSD MSA 아키텍처 개요

## 1. 전환 배경

기존 SSD 서버는 멀티모듈 모놀리식 구조로 구성되어 있다. 이 구조는 도메인 분리가 잘 되어 있다는 장점이 있지만, 배포 단위가 하나이기 때문에 다음과 같은 한계가 있다.

- 특정 기능만 독립적으로 배포하기 어렵다.
- AI 연동처럼 응답 시간이 긴 기능과 일반 CRUD 기능을 같은 런타임에 묶어 운영하게 된다.
- 인증, 문서, 평가, AI 기능이 모두 하나의 애플리케이션 수명주기를 공유한다.
- 쿠버네티스 기반 확장과 장애 격리를 설명하기 어렵다.

이번 전환은 단순히 쿠버네티스에 올리는 것을 넘어서, 서비스 경계를 분리하고 EKS에 적합한 운영 구조를 설계하는 것을 목표로 한다.

## 2. 대상 서비스

이번 1차 설계에서는 아래 6개 서비스를 기준으로 한다.

### 2.1 API Gateway

- 외부 클라이언트의 단일 진입점
- 경로 기반 라우팅
- 공통 헤더 전달
- CORS 정책 처리
- 내부 서비스 헬스 상태 집계의 출발점

### 2.2 Auth Service

- JWT 발급 및 검증
- OAuth 로그인 연동
- 인증 세션 및 토큰 정책 관리

### 2.3 Member Service

- 회원 조회 및 기본 프로필 관리
- 인증 이후 사용자 기본 정보 제공

### 2.4 Document Service

- 문서 생성, 수정, 조회
- 폴더 및 문서 소유권 검증
- 문서 메타데이터와 저장 경계 관리

### 2.5 Review Service

- 평가, 체크리스트, 요약 결과 조회/저장
- 문서에 연결된 리뷰 관점 데이터 관리

### 2.6 AI Orchestrator

- 외부 AI 서버 연동
- 요약, 키워드, 평가 요청 오케스트레이션
- 장기적으로 재시도, timeout, fallback 정책이 집중될 서비스

## 3. 인프라 구성 원칙

- AWS 리전: `ap-northeast-2`
- 환경: `dev`
- 기존 VPC 재사용
- 기존 PostgreSQL RDS 재사용
- EKS는 새로 생성
- ECR은 서비스별 리포지토리로 확장
- ALB Ingress Controller를 사용해 외부 HTTP 트래픽 진입
- 민감 정보는 Secrets Manager와 External Secrets Operator를 사용해 주입

## 4. 네트워크 흐름

1. 사용자가 ALB 엔드포인트로 요청을 보낸다.
2. ALB가 Kubernetes Ingress 규칙에 따라 `ssd-api-gateway`로 요청을 전달한다.
3. API Gateway가 요청 경로를 보고 내부 서비스로 프록시한다.
4. 내부 서비스는 필요한 경우 기존 RDS 또는 외부 AI 서버와 통신한다.

## 5. 데이터 전략

이번 단계에서는 서비스별 물리 DB 분리 대신 공유 RDS를 재사용한다. 다만 설계 원칙은 다음과 같다.

- 서비스별 스키마 분리 가능성을 유지한다.
- 서비스가 직접 다른 서비스의 테이블을 수정하는 구조는 지양한다.
- 장기적으로 서비스 경계가 안정되면 DB 분리 가능하도록 문서화한다.

## 6. 기대 효과

- 서비스별 독립 배포 구조 확보
- AI 처리 기능의 분리 용이성 확보
- EKS 기반 운영 흐름 경험 확보
- 과제와 실무형 아키텍처 사이의 연결 설명 가능
